# 真实业务案例口播视频标准块

这个标准块用于中文抖音/短视频里的真实业务案例、实操经验、合规坑、客户故事、经营风险类口播。尤其适用于没有真实截图、没有聊天记录、没有后台权限、不能伪造官方页面的情况。

## 核心原则

不要做成 PPT。不要伪造截图。

这类视频要做成“口播驱动的业务机制动画”：

- 真实音频和真实字幕决定时间轴。
- 全片只选一个主业务对象作为视觉锚点。
- 每一句话都让这个对象发生一个可见的状态变化。
- 账号、权限、归属、流程、风险、控制权这些抽象概念，必须变成观众看得懂的界面状态、钥匙、阻断、字段、权限流。
- 封面用生图模型做强视觉，不要用一堆 HTML 卡片拼封面。

## 适用场景

用户讲的是下面这类内容时，用这个标准块：

- 客户案例、服务案例、真实业务故事。
- 合规坑、证照坑、平台规则坑。
- 许可证、认证、进出口、税务、法律、账号、后台权限问题。
- 只有电话沟通，没有聊天截图的案例。
- 真实业务流程，但没有可用截图，也不应该伪造截图。

典型例子：

- 客户的 FDA 进口商许可证办下来了，但后台账号绑定在原中介员工名下。
- 证书确实办好了，但公司自己登录不了系统。
- 客户想换服务商，结果发现原来的企业负责人不配合变更。

## 输入

需要拿到或先推导出：

- 最终口播音频，或者已经确认的口播文案。
- 目标平台，默认抖音竖屏 9:16。
- 主业务对象：许可证、证书、账号、订单、合同、后台、表单、审批、产品库等。
- 风险对象：账号控制权、权限持有人、审批阻断、隐藏操作人、密码缺失、名义归属和实际控制不一致等。

如果已经有真实音频，必须先转写音频。后续字幕和时间轴以音频转写为准，不以原文案为准。

## 输出

最终要产出：

- 基于真实音频的逐句字幕时间轴。
- 逐句视觉状态分镜表。
- 用代码图形做出来的机制动画视频。
- 生图模型生成的封面。
- 最终视频：封面可以作为单独封面图，也可以放在视频第一帧，按用户发布习惯决定。

## 制作流程

### 1. 先转写真实音频

- 先把音频转成逐句时间轴。
- 可以修明显识别错误，但不能把口播改写成总结稿。
- 字幕必须贴真实声音。
- 原文案只能当参考，真实录音才是最终标准。

### 2. 找到全片的主业务对象

只选一个可视化对象贯穿全片，例如：

- 一张许可证卡片。
- 一个后台登录面板。
- 一个账号控制权面板。
- 一张订单卡。
- 一个审批表。
- 一个“企业负责人”字段。

不要每句话都新建一堆随机卡片。观众要感觉自己一直在看同一个业务对象一步步出问题。

### 3. 按业务逻辑拆阶段

不要按固定 15 秒硬切。按业务含义拆：

- 案例钩子。
- 现有状态：证已经办好。
- 隐藏问题：谁能登录。
- 关键概念：企业负责人/管理员/账号控制权。
- 冲突：原中介不给账号或不配合。
- 名义归属和实际控制分离。
- 流程被卡住。
- 风险总结。
- 正确做法。
- 结尾提醒清单。

### 4. 把抽象风险做成可见画面

常用映射：

- 账号：登录框、空密码、钥匙令牌。
- 控制权：钥匙在不同角色之间移动。
- 企业负责人：表单里的高亮字段。
- 权限错配：左边是“名义属于 A 公司”，右边是“实际控制在 C 员工”。
- 换服务商：流程按钮变灰、流程线被挡住。
- 拒绝配合：深色回复卡，只突出一句红色重点。
- 正确做法：绿色清单 + 红色划掉错误选项。

### 5. 每个关键词都要有微动效

每一句重要话，至少对应一个视觉状态变化。

关键词触发的动作可以是：

- 出现。
- 聚焦。
- 高亮。
- 放大轻弹。
- 勾选。
- 打叉。
- 划掉。
- 置灰。
- 钥匙转移。
- 红色警示脉冲。
- 卡片覆盖旧状态。

不要让一个静态画面撑 5 到 10 秒，除非这句话本身是在解释一个复杂概念，观众需要读。

### 6. 画面要比声音慢一点点

- 字幕严格跟真实音频走。
- 画面动效通常要比声音慢 `0.3s-0.6s`，这样不会显得画面抢在嘴前面。
- 中文快口播可以先用 `0.55s` 作为视觉延迟。
- 如果某一段单独偏慢或偏快，只调那一段，不要整体乱移音频。

### 7. 封面用生图模型做

不要用密密麻麻的 HTML UI 截图做封面。

封面应该是一个强视觉隐喻：

- 老板拿着证，但钥匙在别人手里。
- 文件在前景，背后有隐藏控制人。
- 证办下来了，但登录令牌不在自己手上。

封面只保留：

- 一个大标题。
- 一个小风险标签。
- 一个清楚的主体画面。

底部 20% 尽量干净，避免被抖音头像、昵称、文案挡住。不要生成假的官方 logo、印章、网站、证书细节。

### 8. 封面进入发布流程

- 如果用户要平台封面，单独导出 PNG。
- 如果用户要第一帧封面，只替换第 0 帧，或者加极短封面帧，不要导致音频整体错位。
- 插入封面后必须重新检查时长和音画同步。

## 分镜表要求

每一行必须有：

- 时间范围：来自真实音频。
- 真实口播句子。
- 字幕文本。
- 业务阶段。
- 主视觉对象。
- 关键词。
- 视觉状态变化。
- 动效触发时间。
- 要抽查的验收帧。

不要写这种空话：

- “展示相关图形”
- “加关键词动画”
- “展示机制”
- “做得更好看”

要写成可执行动作：

- “说到‘系统账号’时，登录框出现蓝色聚焦环。”
- “说到‘员工信息’时，深色拒绝卡里的这四个字变红并轻微放大。”
- “说到‘真正能登录’时，蓝色钥匙从 A 公司移动到 C 员工。”

## 抖音竖屏布局规则

- 主视觉放在上半区和中部安全区。
- 字幕放在下方平台遮挡区之上。
- 底部 20%-25% 不放重要信息。
- 标题和进度条要小，不要抢画面。
- 不要出现两个进度条。
- 主视觉和字幕距离不能太远，否则音画割裂。
- 每个浮层都要有父级归属，不要到处飘。

## 字幕规则

- 字幕必须是口播原话，不是总结。
- 可以修转写错字，但不能润色成新文案。
- 快口播可以一句拆成多个短字幕。
- 长字幕要换行或缩小，不能顶边、压线、撞卡片。
- 主视觉上的文字只放短锚点，不要重复整句字幕。

## 动效验收

最终导出前至少抽查：

- 每个阶段的第一帧。
- 每个阶段信息最密的帧。
- 用户指出过的碰撞时间点。
- 每个关键词动效触发点。
- 最后的提醒清单。
- 如果插入封面，检查第一帧。

检查问题：

- 画面是不是跑在声音前面？
- 画面是不是太晚？
- 字幕是不是和声音原话一致？
- 卡片有没有左右居中？
- 同类模块内部有没有上下居中？
- 底部抖音安全区是否干净？
- 第一帧是不是生图封面？

## 禁止事项

- 不要编造聊天截图。
- 不要编造官方后台截图。
- 不要生成看起来像真实官方证书、印章、网站、背书的图。
- 不要用堆满 UI 卡片的 HTML 画面当封面。
- 不要把真实口播字幕改成总结稿。
- 不要后半段变成静态 PPT。
- 不要用一个通用模板糊完整个视频，导致后面每段长得一样。
- 不要为了同步去移动整条音频，除非用户明确要求。

## 60-120 秒复用结构

可复用结构：

1. 生图封面：强标题 + 风险隐喻。
2. 0-3 秒：案例钩子。
3. 3-10 秒：主业务对象出现。
4. 10-25 秒：隐藏问题和关键概念。
5. 25-45 秒：冲突或拒绝配合。
6. 45-65 秒：机制解释。
7. 65-80 秒：风险总结。
8. 80-90 秒：正确做法。
9. 90 秒以后：最终提醒清单。

一句话原则：

真实业务口播可以没有真实截图，但必须把看不见的业务机制做成“对象状态 + 权限流 + 关键词动效”。
